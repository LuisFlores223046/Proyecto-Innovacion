\hypertarget{dir_47fe9814a2b30dae49bba247dfcf51dd_autotoc_md521}{}\doxysubsection{\texorpdfstring{minimatch}{minimatch}}\label{dir_47fe9814a2b30dae49bba247dfcf51dd_autotoc_md521}
A minimal matching utility.

This is the matching library used internally by npm.

It works by converting glob expressions into Java\+Script {\ttfamily Reg\+Exp} objects.\hypertarget{dir_47fe9814a2b30dae49bba247dfcf51dd_autotoc_md522}{}\doxysubsection{\texorpdfstring{Important Security Consideration!}{Important Security Consideration!}}\label{dir_47fe9814a2b30dae49bba247dfcf51dd_autotoc_md522}
\begin{DoxyWarning}{Warning}
This library uses Java\+Script regular expressions. Please read the following warning carefully, and be thoughtful about what you provide to this library in production systems.
\end{DoxyWarning}
{\itshape Any} library in Java\+Script that deals with matching string patterns using regular expressions will be subject to \href{https://owasp.org/www-community/attacks/Regular_expression_Denial_of_Service_-_ReDoS}{\texttt{Re\+DoS}} if the pattern is generated using untrusted input.

Efforts have been made to mitigate risk as much as is feasible in such a library, providing maximum recursion depths and so forth, but these measures can only ultimately protect against accidents, not malice. A dedicated attacker can {\itshape always} find patterns that cannot be defended against by a bash-\/compatible glob pattern matching system that uses Java\+Script regular expressions.

To be extremely clear:\+

\begin{DoxyWarning}{Warning}
{\bfseries{If you create a system where you take user input, and use that input as the source of a Regular Expression pattern, in this or any extant glob matcher in Java\+Script, you will be pwned.}}
\end{DoxyWarning}
A future version of this library {\itshape may} use a different matching algorithm which does not exhibit backtracking problems. If and when that happens, it will likely be a sweeping change, and those improvements will {\bfseries{not}} be backported to legacy versions.

In the near term, it is not reasonable to continue to play whack-\/a-\/mole with security advisories, and so any future Re\+DoS reports will be considered "{}working as intended"{}, and resolved entirely by this warning.\hypertarget{dir_47fe9814a2b30dae49bba247dfcf51dd_autotoc_md523}{}\doxysubsection{\texorpdfstring{Usage}{Usage}}\label{dir_47fe9814a2b30dae49bba247dfcf51dd_autotoc_md523}

{\bfseries{Please only use forward-\/slashes in glob expressions.}}

Though windows uses either {\ttfamily /\+} or {\ttfamily \textbackslash{}} as its path separator, only {\ttfamily /\+} characters are used by this glob implementation. You must use forward-\/slashes {\bfseries{only}} in glob expressions. Back-\/slashes in patterns will always be interpreted as escape characters, not path separators.

Note that {\ttfamily \textbackslash{}} or {\ttfamily /\+} {\itshape will} be interpreted as path separators in paths on Windows, and will match against {\ttfamily /\+} in glob expressions.

So just always use {\ttfamily /\+} in patterns.\hypertarget{dir_47fe9814a2b30dae49bba247dfcf51dd_autotoc_md526}{}\doxysubsubsection{\texorpdfstring{UNC Paths}{UNC Paths}}\label{dir_47fe9814a2b30dae49bba247dfcf51dd_autotoc_md526}
On Windows, UNC paths like {\ttfamily /\+/\+?/\+c:\+/\+...} or {\ttfamily /\+/\+\+Computer\+Name/\+\+Share/\+...} are handled specially.


\begin{DoxyItemize}
\item Patterns starting with a double-\/slash followed by some non-\/slash characters will preserve their double-\/slash. As a result, a pattern like {\ttfamily /\+/\+\texorpdfstring{$\ast$}{*}} will match {\ttfamily /\+/\+x}, but not {\ttfamily /\+x}.
\item Patterns staring with {\ttfamily /\+/\+?/\+\texorpdfstring{$<$}{<}drive letter\texorpdfstring{$>$}{>}:\+} will {\itshape not} treat the {\ttfamily ?} as a wildcard character. Instead, it will be treated as a normal string.
\item Patterns starting with {\ttfamily /\+/\+?/\+\texorpdfstring{$<$}{<}drive letter\texorpdfstring{$>$}{>}:\+/\+...} will match file paths starting with {\ttfamily \texorpdfstring{$<$}{<}drive letter\texorpdfstring{$>$}{>}:\+/\+...}, and vice versa, as if the {\ttfamily /\+/\+?/\+} was not present. This behavior only is present when the drive letters are a case-\/insensitive match to one another. The remaining portions of the path/\+pattern are compared case sensitively, unless {\ttfamily nocase:\+true} is set.
\end{DoxyItemize}

A number indicating the level of optimization that should be done to the pattern prior to parsing and using it for matches.

Globstar parts {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}} are always converted to {\ttfamily \texorpdfstring{$\ast$}{*}} when {\ttfamily noglobstar} is set, and multiple adjacent {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}} parts are converted into a single {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}} (ie, {\ttfamily a/\+\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}/\+\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}/\+b} will be treated as {\ttfamily a/\+\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}/\+b}, as this is equivalent in all cases).


\begin{DoxyItemize}
\item {\ttfamily 0} -\/ Make no further changes. In this mode, {\ttfamily .} and {\ttfamily ..} are maintained in the pattern, meaning that they must also appear in the same position in the test path string. Eg, a pattern like {\ttfamily a/\+\texorpdfstring{$\ast$}{*}/\+../\+c} will match the string {\ttfamily a/\+b/\+../\+c} but not the string {\ttfamily a/\+c}.
\item {\ttfamily 1} -\/ (default) Remove cases where a double-\/dot {\ttfamily ..} follows a pattern portion that is not {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}}, {\ttfamily .}, {\ttfamily ..}, or empty {\ttfamily \textquotesingle{}\textquotesingle{}}. For example, the pattern {\ttfamily ./\+a/\+b/\+../\+\texorpdfstring{$\ast$}{*}} is converted to {\ttfamily ./\+a/\+\texorpdfstring{$\ast$}{*}}, and so it will match the path string {\ttfamily ./\+a/\+c}, but not the path string {\ttfamily ./\+a/\+b/\+../\+c}. Dots and empty path portions in the pattern are preserved.
\item {\ttfamily 2} (or higher) -\/ Much more aggressive optimizations, suitable for use with file-\/walking cases:\+
\begin{DoxyItemize}
\item Remove cases where a double-\/dot {\ttfamily ..} follows a pattern portion that is not {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}}, {\ttfamily .}, or empty {\ttfamily \textquotesingle{}\textquotesingle{}}. Remove empty and {\ttfamily .} portions of the pattern, where safe to do so (ie, anywhere other than the last position, the first position, or the second position in a pattern starting with {\ttfamily /\+}, as this may indicate a UNC path on Windows).
\item Convert patterns containing {\ttfamily \texorpdfstring{$<$}{<}pre\texorpdfstring{$>$}{>}/\+\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}/\+../\+\texorpdfstring{$<$}{<}p\texorpdfstring{$>$}{>}/\+\texorpdfstring{$<$}{<}rest\texorpdfstring{$>$}{>}} into the equivalent {\ttfamily \texorpdfstring{$<$}{<}pre\texorpdfstring{$>$}{>}/\+\{..,\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}\}/\+\texorpdfstring{$<$}{<}p\texorpdfstring{$>$}{>}/\+\texorpdfstring{$<$}{<}rest\texorpdfstring{$>$}{>}}, where {\ttfamily \texorpdfstring{$<$}{<}p\texorpdfstring{$>$}{>}} is a a pattern portion other than {\ttfamily .}, {\ttfamily ..}, {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}}, or empty {\ttfamily \textquotesingle{}\textquotesingle{}}.
\item Dedupe patterns where a {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}} portion is present in one and omitted in another, and it is not the final path portion, and they are otherwise equivalent. So {\ttfamily \{a/\+\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}/\+b,a/\+b\}} becomes {\ttfamily a/\+\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}/\+b}, because {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}} matches against an empty path portion.
\item Dedupe patterns where a {\ttfamily \texorpdfstring{$\ast$}{*}} portion is present in one, and a non-\/dot pattern other than {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}}, {\ttfamily .}, {\ttfamily ..}, or {\ttfamily \textquotesingle{}\textquotesingle{}} is in the same position in the other. So {\ttfamily a/\+\{\texorpdfstring{$\ast$}{*},x\}/\+b} becomes {\ttfamily a/\+\texorpdfstring{$\ast$}{*}/\+b}, because {\ttfamily \texorpdfstring{$\ast$}{*}} can match against {\ttfamily x}.
\end{DoxyItemize}

While these optimizations improve the performance of file-\/walking use cases such as \href{http://npm.im/glob}{\texttt{glob}} (ie, the reason this module exists), there are cases where it will fail to match a literal string that would have been matched in optimization level 1 or 0.

Specifically, while the {\ttfamily Minimatch.\+match()} method will optimize the file path string in the same ways, resulting in the same matches, it will fail when tested with the regular expression provided by {\ttfamily Minimatch.\+make\+Re()}, unless the path string is first processed with {\ttfamily minimatch.\+level\+Two\+File\+Optimize()} or similar.

Defaults to the value of {\ttfamily process.\+platform}.\hypertarget{dir_47fe9814a2b30dae49bba247dfcf51dd_autotoc_md556}{}\doxysubsubsection{\texorpdfstring{max\+Globstar\+Recursion}{max\+Globstar\+Recursion}}\label{dir_47fe9814a2b30dae49bba247dfcf51dd_autotoc_md556}
Max number of non-\/adjacent {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}} patterns to recursively walk down.

The default of {\ttfamily 200} is almost certainly high enough for most purposes, and can handle absurdly excessive patterns.

If the limit is exceeded (which would require very excessively long patterns and paths containing lots of {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}} patterns!), then it is treated as non-\/matching, even if the path would normally match the pattern provided.

That is, this is an intentional false negative, deemed an acceptable break in correctness for security and performance.\hypertarget{dir_47fe9814a2b30dae49bba247dfcf51dd_autotoc_md557}{}\doxysubsubsection{\texorpdfstring{max\+Extglob\+Recursion}{max\+Extglob\+Recursion}}\label{dir_47fe9814a2b30dae49bba247dfcf51dd_autotoc_md557}
Max depth to traverse for nested extglobs like {\ttfamily \texorpdfstring{$\ast$}{*}(a\texorpdfstring{$\vert$}{|}b\texorpdfstring{$\vert$}{|}c)}

Default is 2, which is quite low, but any higher value swiftly results in punishing performance impacts. Note that this is {\itshape not} relevant when the globstar types can be safely coalesced into a single set.

For example, {\ttfamily \texorpdfstring{$\ast$}{*}(a\texorpdfstring{$\vert$}{|}@(b\texorpdfstring{$\vert$}{|}c)\texorpdfstring{$\vert$}{|}d)} would be flattened into {\ttfamily \texorpdfstring{$\ast$}{*}(a\texorpdfstring{$\vert$}{|}b\texorpdfstring{$\vert$}{|}c\texorpdfstring{$\vert$}{|}d)}. Thus, many common extglobs will retain good performance and never hit this limit, even if they are excessively deep and complicated.

If the limit is hit, then the extglob characters are simply not parsed, and the pattern effectively switches into {\ttfamily noextglob:\+ true} mode for the contents of that nested sub-\/pattern. This will typically {\itshape not} result in a match, but is considered a valid trade-\/off for security and performance.\hypertarget{dir_47fe9814a2b30dae49bba247dfcf51dd_autotoc_md558}{}\doxysubsection{\texorpdfstring{Comparisons to other fnmatch/\+glob implementations}{Comparisons to other fnmatch/\+glob implementations}}\label{dir_47fe9814a2b30dae49bba247dfcf51dd_autotoc_md558}
While strict compliance with the existing standards is a worthwhile goal, some discrepancies exist between minimatch and other implementations. Some are intentional, and some are unavoidable.

If the pattern starts with a {\ttfamily !} character, then it is negated. Set the {\ttfamily nonegate} flag to suppress this behavior, and treat leading {\ttfamily !} characters normally. This is perhaps relevant if you wish to start the pattern with a negative extglob pattern like {\ttfamily !(a\texorpdfstring{$\vert$}{|}B)}. Multiple {\ttfamily !} characters at the start of a pattern will negate the pattern multiple times.

If a pattern starts with {\ttfamily \#}, then it is treated as a comment, and will not match anything. Use {\ttfamily \textbackslash{}\#} to match a literal {\ttfamily \#} at the start of a line, or set the {\ttfamily nocomment} flag to suppress this behavior.

The double-\/star character {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}} is supported by default, unless the {\ttfamily noglobstar} flag is set. This is supported in the manner of bsdglob and bash 4.\+1, where {\ttfamily \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}} only has special significance if it is the only thing in a path part. That is, {\ttfamily a/\+\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}/\+b} will match {\ttfamily a/\+x/\+y/\+b}, but {\ttfamily a/\+\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}b} will not.

If an escaped pattern has no matches, and the {\ttfamily nonull} flag is set, then minimatch.\+match returns the pattern as-\/provided, rather than interpreting the character escapes. For example, {\ttfamily minimatch.\+match(\+[]\+, "{}\textbackslash{}\textbackslash{}\texorpdfstring{$\ast$}{*}a\textbackslash{}\textbackslash{}?"{})} will return {\ttfamily "{}\textbackslash{}\textbackslash{}\texorpdfstring{$\ast$}{*}a\textbackslash{}\textbackslash{}?"{}} rather than {\ttfamily "{}\texorpdfstring{$\ast$}{*}a?"{}}. This is akin to setting the {\ttfamily nullglob} option in bash, except that it does not resolve escaped pattern characters.

If brace expansion is not disabled, then it is performed before any other interpretation of the glob pattern. Thus, a pattern like {\ttfamily +(a\texorpdfstring{$\vert$}{|}\{b),c)\}}, which would not be valid in bash or zsh, is expanded {\bfseries{first}} into the set of {\ttfamily +(a\texorpdfstring{$\vert$}{|}b)} and {\ttfamily +(a\texorpdfstring{$\vert$}{|}c)}, and those patterns are checked for validity. Since those two are valid, matching proceeds.

Negated extglob patterns are handled as closely as possible to Bash semantics, but there are some cases with negative extglobs which are exceedingly difficult to express in a Java\+Script regular expression. In particular the negated pattern {\ttfamily \texorpdfstring{$<$}{<}start\texorpdfstring{$>$}{>}!(\texorpdfstring{$<$}{<}pattern\texorpdfstring{$>$}{>}\texorpdfstring{$\ast$}{*}\texorpdfstring{$\vert$}{|})\texorpdfstring{$\ast$}{*}} will in bash match anything that does not start with {\ttfamily \texorpdfstring{$<$}{<}start\texorpdfstring{$>$}{>}\texorpdfstring{$<$}{<}pattern\texorpdfstring{$>$}{>}}. However, {\ttfamily \texorpdfstring{$<$}{<}start\texorpdfstring{$>$}{>}!(\texorpdfstring{$<$}{<}pattern\texorpdfstring{$>$}{>}\texorpdfstring{$\ast$}{*})\texorpdfstring{$\ast$}{*}} {\itshape will} match paths starting with {\ttfamily \texorpdfstring{$<$}{<}start\texorpdfstring{$>$}{>}\texorpdfstring{$<$}{<}pattern\texorpdfstring{$>$}{>}}, because the empty string can match against the negated portion. In this library, {\ttfamily \texorpdfstring{$<$}{<}start\texorpdfstring{$>$}{>}!(\texorpdfstring{$<$}{<}pattern\texorpdfstring{$>$}{>}\texorpdfstring{$\ast$}{*}\texorpdfstring{$\vert$}{|})\texorpdfstring{$\ast$}{*}} will {\itshape not} match any pattern starting with {\ttfamily \texorpdfstring{$<$}{<}start\texorpdfstring{$>$}{>}}, due to a difference in precisely which patterns are considered "{}greedy"{} in Regular Expressions vs bash path expansion. This may be fixable, but not without incurring some complexity and performance costs, and the trade-\/off seems to not be worth pursuing.

Note that {\ttfamily fnmatch(3)} in libc is an extremely naive string comparison matcher, which does not do anything special for slashes. This library is designed to be used in glob searching and file walkers, and so it does do special things with {\ttfamily /\+}. Thus, {\ttfamily foo\texorpdfstring{$\ast$}{*}} will not match {\ttfamily foo/\+bar} in this library, even though it would in {\ttfamily fnmatch(3)}.
