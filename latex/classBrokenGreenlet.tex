Perform a stack switch into this greenlet.

This temporarily sets the global variable {\ttfamily switching\+\_\+thread\+\_\+state} to this greenlet; as soon as the call to {\ttfamily slp\+\_\+switch} completes, this is reset to NULL. Consequently, this depends on the GIL.

TODO:\+ Adopt the stackman model and pass {\ttfamily slp\+\_\+switch} a callback function and context pointer; this eliminates the need for global variables altogether.

Because the stack switch happens in this function, this function can\textquotesingle{}t use its own stack (local) variables, set before the switch, and then accessed after the switch.

Further, you con\textquotesingle{}t even access {\ttfamily g\+\_\+thread\+\_\+state\+\_\+global} before and after the switch from the global variable. Because it is thread local some compilers cache it in a register/\+on the stack, notably new versions of MSVC; this breaks with strange crashes sometime later, because writing to anything in {\ttfamily g\+\_\+thread\+\_\+state\+\_\+global} after the switch is actually writing to random memory. For this reason, we call a non-\/inlined function to finish the operation. (XXX:\+ The {\ttfamily /\+\+GT} MSVC compiler argument probably fixes that.)

It is very important that stack switch is \textquotesingle{}atomic\textquotesingle{}, i.\+e. no calls into other Python code allowed (except very few that are safe), because global variables are very fragile. (This should no longer be the case with thread-\/local variables.)
